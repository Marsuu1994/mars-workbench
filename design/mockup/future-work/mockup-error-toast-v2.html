Design notes — Pattern A error toast (see design/spike/error-handling.md)

  • Placement: fixed top-center, below the page header, on both breakpoints — desktop centers over the board/matrix area; mobile reuses the existing success-toast slot (top-[calc(env(safe-area-inset-top)+7rem)]). The success toast is mobile-only today and would adopt this same slot on desktop.
  • Surface: rounded-[10px] border-error/30 bg-error/15 text-error backdrop-blur-sm shadow-lg at z-50 — above sheet backdrops, so a failure surfaced from the mobile backlog sheet stays visible; the blur keeps board content legible beneath. Never a filled red block.
  • Auto-dismiss: 5000 ms (ERROR_TOAST_DURATION_MS), deliberately longer than the success toast's 2500 ms TOAST_DURATION_MS; a 2px bg-error/40 countdown bar telegraphs it; hover / touch-hold pauses the timer; manual ✕ always available.
  • Replace, don't stack: one toast at a time — a newer error replaces the current one. These quick ops are serial in practice; a queue is overkill.
  • Snap-back marker: the rolled-back card briefly wears a border-error/40 outline so the failure is attributable to the card that snapped back.
  • Wired call sites (the four optimistic helpers whose failure branch currently only does rollback + console.error): KanbanBoard.handleDragEnd, KanbanBoard.handlePullToTodo, PriorityMatrixPage reprioritize (handleDragEnd) and handleTrack.
  • Copy / i18n: proposed keys Errors.moveFailed (board drag + pull to Todo), Errors.reprioritizeFailed, Errors.trackFailed; the actionable variant reuses the existing Errors.noActivePlan plus a "Create plan" action linking /kanban/plans/new — specific copy only when it's user-actionable, generic otherwise (spike open question).
  • Plumbing: a toastStore (Zustand, per-domain-store convention) + one AppToaster mounted in the application layer; helpers push a toast in the same branch that rolls back. No error object ever crosses the wire — generic outward, specific inward.
